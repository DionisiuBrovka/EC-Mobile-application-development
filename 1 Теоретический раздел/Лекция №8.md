# Пакеты и интерфейсы

## Цели и задачи лекции
- Понять роль пакетов в организации кода на Python.
- Ознакомиться с механизмом создания и использования пакетов.
- Изучить концепцию интерфейсов и их применение в Python.
- Научиться документировать и тестировать пакеты и интерфейсы.

## Ключевые понятия
| Термин | Определение |
|--------|-------------|
| **Пакет** | Набор модулей, объединённых в директорию с файлом `__init__.py`. |
| **Модуль** | Файл с расширением `.py`, содержащий код. |
| **Импорт** | Операция включения кода из другого модуля или пакета. |
| **Интерфейс** | Набор методов и свойств, которые класс должен реализовать. |
| **Абстрактный класс** | Класс, содержащий абстрактные методы, определяемые в `abc` модуле. |
| **Пакет-namespace** | Пакет без `__init__.py`, позволяющий объединять модули с одинаковыми именами. |
| **Пакет-namespace** | Пакет без `__init__.py`, позволяющий объединять модули с одинаковыми именами. |
| **Конвенция `__all__`** | Список публичных объектов, экспортируемых при `from package import *`. |
| **Файл `setup.py`** | Скрипт, описывающий пакет для сборки и распространения. |
| **PEP 8** | Руководство по стилю кода Python. |

## Основное содержание

### 1. Организация кода в пакеты (30 мин)
#### 1.1 Что такое пакет и зачем он нужен
- Деление больших проектов на модули.
- Избежание конфликтов имён.
- Упрощение переиспользования кода.

#### 1.2 Структура пакета
```
my_package/
├─ __init__.py
├─ module_a.py
├─ module_b.py
└─ subpackage/
   ├─ __init__.py
   └─ module_c.py
```
- `__init__.py` может быть пустым, но обязателен для Python 3.5–до 3.11.
- В Python 3.11 можно использовать namespace пакеты.

#### 1.3 Импорт из пакетов
```python
# Внутри my_package/module_a.py
from .module_b import func_b

# В другом месте проекта
from my_package import module_a
```

### 2. Создание собственного пакета (30 мин)
#### 2.1 Инициализация `__init__.py`
```python
# my_package/__init__.py
__all__ = ['module_a', 'module_b']
```

#### 2.2 Сборка и установка пакета
```bash
python setup.py sdist bdist_wheel
pip install .
```

#### 2.3 Пример `setup.py`
```python
from setuptools import setup, find_packages

setup(
    name='my_package',
    version='0.1.0',
    packages=find_packages(),
    install_requires=['requests'],
    python_requires='>=3.8',
)
```

### 3. Интерфейсы и абстракции (30 мин)
#### 3.1 Абстрактные базовые классы (ABC)
```python
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self):
        pass
```

#### 3.2 Реализация интерфейса
```python
class Rectangle(Shape):
    def __init__(self, width, height):
        self.width = width
        self.height = height

    def area(self):
        return self.width * self.height
```

#### 3.3 Проверка реализации
```python
>>> isinstance(Rectangle(3, 4), Shape)
True
```

#### 3.4 Плюсы и минусы использования ABC
- Чёткая контрактная спецификация.
- Возможность статической проверки с помощью `mypy`.
- Избыточная сложность для простых проектов.

### 4. Тестирование и документация (30 мин)
#### 4.1 Тесты для пакетов
```python
# tests/test_rectangle.py
from my_package.module_a import Rectangle

def test_area():
    r = Rectangle(5, 6)
    assert r.area() == 30
```

#### 4.2 Документация с помощью `pydoc` и `Sphinx`
```bash
pydoc -w my_package.module_a
```

#### 4.3 Конвенции именования и PEP 8
- Имена пакетов: маленькие буквы, без подчеркивания.
- Имена модулей: маленькие буквы, без пробелов.

## Примеры и иллюстрации

### Пример пакета «math_utils»
```
math_utils/
├─ __init__.py
├─ algebra.py
├─ calculus.py
└─ tests/
   └─ test_algebra.py
```

#### `algebra.py`
```python
def add(a, b):
    return a + b

def mul(a, b):
    return a * b
```

#### `tests/test_algebra.py`
```python
from math_utils.algebra import add, mul

def test_add():
    assert add(2, 3) == 5

def test_mul():
    assert mul(4, 5) == 20
```

### Схема импорта
```
project/
├─ main.py
└─ utils/
   ├─ __init__.py
   └─ helpers.py
```
```python
# main.py
from utils.helpers import helper_func
```

## Выводы по лекции
- Пакеты позволяют структурировать код, избегать конфликтов имён и облегчать переиспользование.
- Создание пакета требует `__init__.py`, `setup.py` и соблюдения конвенций.
- Интерфейсы реализуются через абстрактные базовые классы, обеспечивая контрактную спецификацию.
- Тестирование и документация являются неотъемлемой частью разработки пакетов.
